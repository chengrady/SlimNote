# SlimNote 优化清单

## 文档说明

这份清单按优先级排序，目标不是把功能做多，而是把 SlimNote 打磨成一个定位清晰、可长期使用的本地工程文本工作台。

建议原则：

- 优先做高频刚需，而不是低频炫技功能。
- 优先补齐工作流闭环，而不是分散堆点状能力。
- 优先强化 SlimNote 的差异化，不盲目追逐 Obsidian 的知识库路线。

---

## P0：优先级最高

### 1. 明确产品定位

目标：让用户在第一次打开时就知道 SlimNote 擅长什么。

建议动作：

- 将产品定位明确为“本地工程文本工作台”。
- 首页与 README 聚焦三类核心场景：Markdown 成稿、JSON/接口整理、日志排障。
- 弱化“万能编辑器”的表述，减少功能边界模糊。
- 把欢迎页中的工作流入口继续强化为真正的主导航。

完成标志：

- 新用户在 10 秒内能理解产品核心用途。
- 官网文案、README、欢迎页文案一致。

### 2. 补齐全局搜索与替换

目标：让 SlimNote 从单文件编辑器升级为可处理工作区的工具。

建议动作：

- 支持当前文件搜索与替换。
- 支持目录级全文搜索。
- 支持目录级批量替换。
- 搜索结果支持按文件分组展示。
- 支持正则、大小写、全词匹配等常用选项。

完成标志：

- 用户能在一个目录内完成“查找问题配置并统一修复”的流程。

### 3. 强化工作区能力

目标：让用户围绕“项目”工作，而不是只围绕“单文件”工作。

建议动作：

- 增加固定项目列表。
- 支持最近目录和最近项目的快速恢复。
- 增加工作区级会话恢复。
- 支持多目录切换与快速跳转。
- 为工作区保留独立的 UI 状态和编辑状态。

完成标志：

- 用户再次打开项目后，可以恢复主要工作上下文。

### 4. 深化设置体系

目标：让 SlimNote 从“可用”走向“可长期主用”。

建议动作：

- 支持快捷键自定义。
- 支持按文件类型保存偏好设置。
- 支持自动保存策略设置。
- 支持 Markdown 默认打开模式设置。
- 支持 JSON、SQL、日志工具的独立偏好项。
- 支持工作区级设置与全局设置分层。

完成标志：

- 高级用户可以将编辑器调整到贴合自己习惯的状态。

---

## P1：第二阶段重点

### 5. 把 Markdown 做成真正的优势项

目标：让 Markdown 成为 SlimNote 的核心卖点之一。

建议动作：

- 保证所见即所得、源码模式、预览模式三者排版一致。
- 优化标题、列表、表格、引用、代码块在不同视图下的视觉一致性。
- 优化大文档滚动同步和模式切换性能。
- 提升图片粘贴、富文本复制、PDF 导出的一致性。
- 为长文档增加更稳定的目录定位和折叠体验。

完成标志：

- 用户在不同 Markdown 模式间切换时，没有明显样式漂移或认知断裂。

### 6. 做强 JSON / SQL / Log 三条工程流

目标：把 SlimNote 和普通 Markdown 编辑器区分开。

建议动作：

- JSON：增加校验、修复、格式化、对比、结构树联动的完整链路。
- SQL：增加常用片段、格式化规则、关键字风格切换、方言差异支持。
- Log：增强等级过滤、关键字高亮、时间范围筛选、异常上下文跳转。
- 三类工作流都支持从欢迎页快速进入。

完成标志：

- 用户能直接在 SlimNote 完成日常工程文本处理，而不用频繁切换外部工具。

### 7. 增加批量处理能力

目标：提升效率，而不是只提升单次编辑体验。

建议动作：

- 增加批量格式化。
- 增加目录内 JSON 扫描与修复。
- 增加日志批量导入与聚合查看。
- 增加跨文件导出与汇总能力。

完成标志：

- 一组文件的处理时间明显下降。

---

## P2：中长期建设

### 8. 引入轻量插件机制

目标：为未来扩展留接口，但不要过早做重平台化。

建议动作：

- 先设计内部扩展点，而不是一上来开放完整插件生态。
- 优先支持工具条扩展、右键菜单扩展、文件类型处理扩展。
- 定义最小可用插件协议。
- 保持插件系统轻量、可控、不影响主应用稳定性。

完成标志：

- 新工具能力可以在不重构主界面的前提下接入。

### 9. 加入 Git 与终端能力

目标：减少用户在编辑器和外部工具之间切换。

建议动作：

- 支持查看文件修改状态。
- 支持基础 diff 浏览。
- 支持快速复制当前文件路径、在终端打开目录。
- 如果加终端，先做轻量版本，不追求 IDE 级复杂度。

完成标志：

- 基础代码与文本排障流程可以在 SlimNote 内闭环完成。

### 10. 建立稳定的发布与文档机制

目标：让外部用户更容易判断产品成熟度。

建议动作：

- README 与实际功能保持同步。
- 区分“已完成”“实验性”“规划中”。
- 每个版本输出清晰的更新日志。
- 为核心能力写独立使用说明文档。

完成标志：

- 用户能快速理解当前版本能做什么、适合什么场景。

---

## 两周执行版清单

如果按个人开发节奏推进，建议先做这 8 项：

- 完成 README 与功能现状对齐。
- 为欢迎页补齐更清晰的产品定位文案。
- 落地当前文件搜索与替换。
- 落地目录级搜索。
- 完善最近项目与固定项目。
- 增加工作区会话恢复。
- 增强 Markdown 三视图样式一致性。
- 为设置页补充自动保存与默认模式选项。

---

## 不建议优先做的事

以下方向暂时不建议抢先做：

- 大而全的知识库系统。
- 复杂插件市场。
- 过重的 AI 功能堆叠。
- 一开始就做完整 IDE 级终端集成。
- 为了追 Obsidian 而做图谱、双链、块引用全套系统。

原因：这些方向投入高、维护重，而且会冲淡 SlimNote 当前最有潜力的差异化路线。

---

## 一句话结论

SlimNote 最值得优化的方向，不是继续横向加功能，而是把“本地工程文本工作台”这条路线做深：工作区能力更强、Markdown 更稳、JSON/SQL/日志更好用、设置更专业、文档更一致。
